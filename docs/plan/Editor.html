Editor — Design Description

A full-page review surface that opens in a new tab after the Recorder hits Complete Capture. The Editor is where a freshly-recorded Guide becomes a publishable document: review the auto-generated descriptions, drop bad Steps, fix the order, set a Title — then Export.

Layout

A three-region frame: a slim toolbar across the top (brand, breadcrumb, save state, Export), a two-column body (a narrow Step rail on the left for navigation, a main editing column on the right), and a sticky footer anchoring Export at the bottom of the viewport. The footer Export and the toolbar Export are intentionally duplicated: the Recorder reads Steps top-down and the bottom button is what's under their thumb when they finish.

The Step rail collapses on narrow viewports (under 1100px). The main column never wider than a comfortable single-column reading width — Steps are reviewed sequentially, not scanned in parallel.

Step row anatomy

Five-column grid per row: drag handle · step number · Annotated Screenshot · description · actions. The Annotated Screenshot is fixed-width (280 px, 16:10) so every row aligns vertically — descriptions sit on a consistent left edge. The description is an inline-edit <textarea>; clicking anywhere on the description text enters edit mode, and an amber focus ring confirms which Step is being edited.

Drag handle and per-Step action buttons (duplicate, delete) reveal on hover — never crowding the row at rest. Per the existing rule in docs/adr/0003-ui-design-decisions.md, an empty rule-based description renders the muted placeholder "Describe this Step"; the row never collapses.

The Amber Accent

Continuing the rule from the Side Panel design: #F59E0B is a signal color, not a brand accent. In the Editor it appears in exactly three roles:

  • The Annotated Screenshot highlight ring (drawn into the captured pixels, not CSS).
  • The primary Export CTA in the top toolbar and the sticky footer.
  • The focus ring on the Step description currently being edited, and the drop-target indicator while drag-reordering.

No amber on the save-state pip, the breadcrumb, the rail, the section labels, or any secondary button. If any of those creep into amber, the Export button stops reading as the one action that ships work.

Drag-to-reorder

Hover any row → drag handle fades in. Mouse-down on the handle lifts the row into a "dragging" treatment: amber outline, slight shadow, opacity 100% while siblings drop to 55%. An amber 2px drop-target indicator (with a leading dot) appears between rows where the cursor is hovering. Releasing commits the new order; pressing Esc cancels.

The Step rail mirrors the same interaction with an Insert here slot, so the Recorder can also use the rail for coarse repositioning across long Guides without scrolling the main column.

Title and Description fields

The Title is an unbordered, oversized input (28px Inter SemiBold) that reads as a heading until focused — the same affordance pattern as Notion. The Description sits directly below as a two-line auto-growing textarea. Both placeholder values ("Untitled Guide", "Add a one-line description…") appear in muted text so the empty state still communicates what the field is for.

A monospace meta row below the Title shows the Step count, recording timestamp, and "last edited" stamp — the only place in the Editor where time-relative copy is allowed.

Auto-save and Export

There is no explicit Save button. Every keystroke debounces a write to IndexedDB; the toolbar pip flips between Saved · just now (green) and Saving… (animated grey) so the Recorder always sees that work is durable. Closing the tab mid-edit is safe.

Export is the only explicit action in the Editor. Clicking it opens a menu listing the four formats with a one-line "what this is good for" hint each — the Recorder shouldn't need to guess what the file extension implies. The Export button is disabled when the Guide has zero Steps (the only invalid export state).

Watch Out For (Implementation Notes for Slice 5)

  • Auto-save races deletion. When the Recorder deletes a Step while a debounced save for that Step's description is in flight, the save can re-insert it. Cancel pending writes by Step ID on delete; reconcile on the IndexedDB transaction layer.
  • Step renumbering on reorder. The Step id stays stable across reorders; the order field on each Step is what changes. The mono "01 / 02 / …" labels are computed from array index, not stored. This matters for the Step rail's "Insert here" indicator — it operates on positions, not IDs.
  • Delete confirmation. Single-click delete with a 5-second undo toast is preferred over a confirm dialog — the Recorder will be deleting many junk Steps in a row and a modal each time is painful. The undo toast must restore the Step's screenshot blob, not just its description.
  • Annotated Screenshot is immutable post-capture. The Editor can delete a Step but cannot re-draw its highlight or replace its screenshot. If the Recorder wants the click in a different spot, they re-record that fragment of the Session. Don't add an "edit screenshot" affordance — it implies capabilities we won't ship.
  • Step rail and main column scroll independently. The rail is sticky to the top of the body; the main column scrolls. Clicking a rail item scroll-anchors the matching Step into view in the main column with a brief amber-tinted highlight flash so the Recorder can confirm the jump.
  • Empty description placeholder must persist after edit-and-clear. If the Recorder types a description and then deletes it back to empty string, the "Describe this Step" placeholder reappears. Don't treat "user emptied" differently from "rule-based produced empty".
  • Export menu position. The menu opens above the footer Export button. Anchor it to the bottom-right of the button's bounding rect and clamp into the viewport — at narrow widths, the 280px menu must not overflow the right edge.
  • Footer storage indicator. Mirrors the Popup's storage warning rule (ADR 0003): if total IndexedDB usage crosses ~80% of quota while the Editor is open, the footer "1.4 MB" stamp recolors amber. Recorder must learn this before Export silently fails, not after.
  • Discard ≠ Delete Guide. The Discard button in the footer trashes the in-progress Guide and returns the Recorder to the Popup. It does not remove already-Exported files. Confirm dialog required — this one is destructive enough to justify the modal.
  • No "Add Step" button anywhere. Steps are produced exclusively by the Recording Session. Manual Step creation is out of scope and adding the affordance would imply the Editor can author screenshots, which it cannot. If the Recorder needs more Steps, they re-record.
