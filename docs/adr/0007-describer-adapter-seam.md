# 0007. Use Adapter Seam for AI Describers

**Date:** 2026-05-23

## Status

Accepted

## Context

The LocalGuide extension supports an opt-in AI description feature. Depending on user preference and configuration, descriptions might be generated by different providers (e.g., Google Gemini via OAuth, OpenAI or Anthropic via Bring-Your-Own-Key). 

As we extract `describer.js` from `background.js` (Slice #18), we need to ensure that the core background logic remains decoupled from the specific implementation details of any single AI provider. We want to avoid polluting the background service worker or the core describer logic with provider-specific API calls, authentication handling, or state management.

## Decision

We will inject the AI provider as an argument to the `describe` function (`describe(metadata, aiProvider)`), rather than having `describer.js` or `background.js` directly import or manage provider logic.

The `describe` function establishes an adapter seam. If an `aiProvider` is provided, it calls `await aiProvider.generateDescription(metadata)`. If this fails or if no provider is present, it falls back to the local `ruleBasedDescription(metadata)`.

## Consequences

*   **Positive:** This decision is load-bearing for the AI descriptions workstream. It creates a clear boundary (adapter seam) between core extension logic and external AI services.
*   **Positive:** It keeps `describer.js` purely functional and highly testable, allowing us to easily inject mock providers during unit tests.
*   **Positive:** The core architecture and background service worker do not need to change when we add new AI services or update existing ones.
*   **Negative:** The caller (`background.js`) is now responsible for initializing and passing the correct `aiProvider` instance, which may require minor state management to know which provider is active.
